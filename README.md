# MNIST Digit Recognition — STM32N6570-DK

Real-time handwritten digit classification (0–9) running entirely on the **STM32N6570-DK** Discovery Kit.  
The IMX335 camera captures a live frame, a custom preprocessing pipeline prepares it for the CNN, and the **Neural-ART NPU** runs inference in under 5 ms. The result is printed over UART and shown on the LCD.

## How It Works

### Inference Pipeline

```
IMX335 camera
  └─ DCMIPP pipe 2: downscale → 32×32 RGB888 → nn_raw (uint8[3072])
       │
       │  Pass 1 — Inverted grayscale (Rec.601)
       │    val = (255 − (0.299·R + 0.587·G + 0.114·B)) / 255
       │    → dark ink becomes bright  (MNIST convention: white digit on black)
       │    → scan all 1024 pixels to find vmin and vmax
       │
       │  Pass 2 — Adaptive background suppression
       │    midpt = (vmin + vmax) / 2
       │    pixel ≤ midpt  →  0.0   (background zeroed)
       │    pixel > midpt  →  (val − midpt) / (vmax − midpt)  rescaled to [0,1]
       │
       │    This step was critical: a simple min-max stretch amplified lighting
       │    gradients in the background, causing the model to hallucinate digits.
       │    The midpoint threshold eliminates that noise completely.
       │
       ▼
  nn_in (float32[32×32×3] = 12 288 bytes, channel-last)
       │
       │  Neural-ART NPU — mnist_cnn_32x32_OE_3_3_1
       │  Weights in NOR flash @ 0x71000000
       │  Inference time: < 5 ms @ 1 GHz
       │
       ▼
  nn_out (float32[10]) — softmax probabilities for digits 0–9
       │
       │  Postprocess: argmax, reject if confidence < 70%
       │
       ▼
  LCD overlay + UART
```

### Model

| Property         | Value                                 |
|------------------|---------------------------------------|
| Name             | `mnist_cnn_32x32_OE_3_3_1`           |
| Framework        | ONNX → STEdgeAI → Neural-ART binary   |
| Input            | 32 × 32 × 3 float32 (RGB, channel-last) |
| Output           | 10-class softmax (digits 0–9)         |
| Inference time   | < 5 ms on Neural-ART NPU @ 1 GHz     |
| Weights location | NOR flash @ `0x71000000`              |

## Workflow

### 1 — Build

Open `Application/STM32N6570-DK/STM32CubeIDE/.project` in **STM32CubeIDE v1.17.0**.  
Build → the IDE produces `STM32CubeIDE/Debug/Project_sign.bin` (post-build signing step is automatic).

### 2 — Flash

Board boot switch must be in **DEV mode** for flashing. From a PowerShell terminal in the project root:

```powershell
# First time (or when FSBL needs updating)
.\flash.ps1

# After a code-only rebuild (faster)
.\flash.ps1 -AppOnly

# When the ONNX model changes (rare)
.\flash.ps1 -Weights
```

> Requires [STM32CubeProgrammer v2.18+](https://www.st.com/en/development-tools/stm32cubeprog.html).  
> After flashing, switch to **BOOT FROM FLASH** and power cycle to run standalone.

### 3 — Monitor

Connect a serial terminal at **115200 baud, 8N1** on USART1 (PE5 TX / PE6 RX).

Live camera output:
```
[2] 94%  (3ms)
[3] 99%  (3ms)
```

TEST_MODE output:
```
digit=3  predicted=3  conf=91%  OK  (3ms)
digit=7  predicted=2  conf=99%  WRONG  (3ms)
```

## TEST_MODE

Set `#define TEST_MODE 1` in [main.c](Application/STM32N6570-DK/Src/main.c) to run inference on 10 synthetic images from [digits.h](Application/STM32N6570-DK/Inc/digits.h) instead of the live camera. Each image is held for `DIGIT_HOLD_MS` ms and the result is printed with OK/WRONG verdict.

This mode is useful to verify the model and preprocessing without needing the physical camera setup.

## Test Results

Three test sessions were conducted. Full data in [Doc/test-results.md](Doc/test-results.md).

### Synthetic images (TEST_MODE)

The synthetic digits use a 7-segment pixel-art style. Digits with smooth curves (0, 2, 3, 5, 6) work well; digits whose shapes require diagonals or loops (7, 8, 9) fail because the rectangular strokes don't match the MNIST training distribution.

**Accuracy: 6/10** after redesigning the failing images.

### Live camera — handwritten digits on paper

Black marker on white A4 paper, 10–15 cm from camera, indoor lighting.

## Camera Tips

- Hold the digit **10–15 cm** from the camera, centred in frame
- **Black marker on white paper**, under uniform ambient light
- Avoid shadows across the digit
- Confidence threshold: predictions below **70%** are rejected and not displayed

---

## Known Limitations

- Digit **7** written freehand is not recognised — handwriting style diverges from MNIST training data.
- Digit **3** handwritten has borderline confidence — thicker strokes and good centering help.
- Digits with similar topology (**2↔3**, **1↔7**) may confuse when partially out of frame or tilted.
- The background suppression assumes the digit is the darkest region in frame — poor results in very low contrast scenes.
